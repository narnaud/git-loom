# Spec 018: Add (Stage Files)

## Overview

`git loom add` stages files into the git index using short IDs, plain
filenames, or the `zz` shorthand for staging everything. It is the loom
equivalent of `git add`, enhanced with short-ID resolution. With `-p`, it
opens a two-pane TUI for interactive hunk-level staging and unstaging.

## Why Add?

Within a loom integration branch, working tree changes are identified by
short IDs in `loom status` output. Using `git add` requires the user to
type full paths. `loom add` lets users stage files using the same compact
short IDs they see in the status display.

The `-p` mode goes further than `git add -p`: it presents both staged and
unstaged hunks in a single view, lets the user toggle each one, and applies
all changes atomically on confirm. This bidirectional model — stage unstaged
hunks, unstage already-staged hunks — eliminates the need to juggle
`git add -p` and `git reset -p` separately.

## CLI

```bash
git-loom add [-p] [<files...>]
```

**Flags:**

- `-p, --patch`: open the interactive hunk selector TUI (see Interactive
  Partial Staging below)

**Arguments:**

- `<files...>`: One or more files to stage. If empty default to interactive mode.
  Accepts:
  - **Short IDs**: compact identifiers from `loom status` output (e.g. `a3`, `0f`)
  - **Plain filenames**: relative paths (e.g. `src/main.rs`)
  - **`zz`**: stages all changes (like `git add -A`)

**Staging behavior:**

- Short IDs and filenames: stages only those files
- `zz`: stages all unstaged changes (equivalent to `git add -A`)
- `zz` mixed with other args: `zz` wins (stages everything)
- No args (without `-p`): error with usage message

## Plain Staging

### What Happens

1. Open the repository.
2. If any argument is `"zz"`, stage all changes and return.
3. Otherwise, resolve each argument via the short-ID resolver:
   - If it matches a file short ID, use the resolved file path
   - If it matches a plain filename in the working tree, use it directly
   - If it matches nothing, error with a descriptive message
4. Stage all resolved files.
5. Print a confirmation: `"Staged N file(s)"`

### Error Cases

| Condition | Message |
|-----------|---------|
| No arguments (without `-p`) | `"No files specified. Use short IDs, filenames, or 'zz' to stage all."` |
| Short ID not found | Standard resolve_arg error: `"Could not resolve '<id>' as a file"` |
| File does not exist | Standard resolve_arg error |

### Examples

```bash
# Stage a single file by short ID
$ loom add a3

# Stage multiple files
$ loom add a3 0f src/lib.rs

# Stage everything
$ loom add zz
```

## Interactive Partial Staging (`-p`)

`loom add -p` opens an interactive TUI for toggling individual hunks between
staged and unstaged states. Unlike `git add -p`, it presents both directions
in a single view: already-staged hunks can be deselected (unstaged) and
unstaged hunks can be selected (staged).

### CLI

```bash
git-loom add -p [<files...>]
```

**Arguments (optional with `-p`):**

- No args: show all changed files (both staged and unstaged)
- `<file>` or `<shortid>`: filter to that specific file's hunks
- `zz`: same as no args (show all changed files)

### What Happens

1. Open the repository and determine the working directory.
2. Enumerate all files with changes (via git status).
3. If specific files are given, resolve via short-ID or filename and filter.
4. For each changed file, collect hunks from both sources:
   - **Staged hunks** (HEAD → index): from `git diff --cached -- <path>`.
     These start as **selected** (reflecting their current staged state).
   - **Unstaged hunks** (index → worktree): from `git diff -- <path>`.
     These start as **not selected** (reflecting their current unstaged state).
5. Sort all hunks within each file by line position so staged and unstaged
   hunks interleave correctly.
6. If no files have hunks, print `"No changes to stage"` and exit.
7. Launch the TUI hunk selector.
8. On exit:
   - **Cancel** (q/Esc/Ctrl+C): print `"Cancelled"` and exit without changes.
   - **Confirm** (c/Enter): apply all selection changes bidirectionally.

### Hunk Origin and Initial Selection

Every hunk carries an origin tag that determines its initial selection state
and the direction of the operation applied on confirm:

| Origin | Initial state | If toggled |
|--------|---------------|------------|
| Staged | Selected | Deselecting causes the hunk to be **unstaged** (reverse-applied from index) |
| Unstaged | Not selected | Selecting causes the hunk to be **staged** (applied to index) |

This means the TUI shows the current state of the index as a starting point
and the user adjusts it to the desired state.

### Special File Types

| File state | Behavior |
|------------|----------|
| Staged deletion (`D` in index) | Single hunk with text `"(file deleted)"`, selected. Deselecting unstages the entire file. |
| Unstaged deletion (`D` in worktree) | Single hunk with text `"(file deleted)"`, not selected. Selecting stages the entire file. |
| Staged new file (`A` in index) | Hunks parsed from `git diff --cached`. Deselecting unstages the entire file. |
| Untracked file (`??`) | Content is read directly from the filesystem. A synthetic diff is built (`@@ -0,0 +1,N @@`). Selecting stages the entire file via `git add`. |
| Empty untracked file | Single hunk with text `"(empty file)"`, not selected. |
| Binary file (staged) | Single hunk with text `"(binary file)"`, selected. Deselecting unstages the entire file. |
| Binary file (unstaged) | Single hunk with text `"(binary file)"`, not selected. Selecting stages the entire file via `git add`. |
| Binary detection (untracked) | Any null byte in the first 8 KB of the file (same heuristic as git). |

### TUI Layout

The TUI uses a two-pane layout with files grouped by directory:

```
╭─ Files ──────────╮╭─ Diff ─────────────────────────╮
│ M  main.rs       ││ [✓] Hunk 1/3 (staged)          │
│ ▼ src/           ││ @@ -10,4 +10,6 @@              │
│   MM lib.rs      ││ -old line                       │
│   A  new.rs      ││ +new line                       │
│ ?? README.md     ││                                 │
╰──────────────────╯╰────────────────────────────────╯
 Navigate: ↑/↓ or j/k | Switch Pane: tab | Toggle: space | Confirm: c or Enter | Quit: q or Esc
```

- **Left pane (~30%):** File list, grouped by directory.
  - Root-level files appear directly.
  - Files in subdirectories are grouped under a `▼ dir/` header.
  - Each file shows its **effective status** — two characters reflecting what
    `git status` would show if the current selections were applied:
    - `M` = fully staged modification
    - `MM` = partially staged (some hunks staged, some not)
    - `M` = fully unstaged modification
    - `A` = new file, staged
    - `??` = untracked (or staged new file fully deselected)
    - `D` / `D` = deletion (unstaged/staged)
  - File names are color-coded: green for fully staged, yellow for partially
    staged, default for unstaged/untracked.
  - The focused file/directory is highlighted with `>`.
- **Right pane (~70%):** Diff view for the focused file. Each hunk has a
  header line `[✓] Hunk 1/3 (staged)` or `[ ] Hunk 2/3`. The `(staged)`
  label appears only for hunks originating from the index. Diff lines are
  color-coded: added lines in green, removed lines in red, context in dim,
  hunk headers in the accent color. Supports vertical scrolling.
- **Directory header selected:** When the cursor is on a directory header,
  the right pane shows `"N file(s) in dir/"` instead of a diff.
- **Status bar:** Shows available keybindings.

### Keyboard Bindings

| Key | Action |
|-----|--------|
| `q` / `Esc` | Quit without changes |
| `c` (without Ctrl) | Confirm — apply selected changes |
| `Enter` | Confirm — apply selected changes |
| `Ctrl+C` | Quit without changes |
| `Tab` / `Shift+Tab` | Switch focus between left and right pane |
| `↑` / `k` | Navigate up (files in left pane, hunks in right pane) |
| `↓` / `j` | Navigate down |
| `Space` | Toggle: in left pane, toggles all hunks in the focused file or directory. In right pane, toggles the focused hunk. |

When toggling a file in the left pane: if any hunk is selected, all hunks
are deselected; if none are selected, all are selected.

When toggling a directory header: the same logic applies across all files in
the directory.

### Staging Mechanism

When the user confirms, the TUI returns the updated file entries. The
changes are then applied in four phases:

1. **Unstage hunks** (reverse-apply): For staged hunks that were deselected,
   build a reverse patch and apply it via `git apply --cached -R`. For
   binary files, staged deletions, and staged new files, unstage the whole
   file via `git reset`.

2. **Stage hunks** (forward-apply): For unstaged hunks that were selected,
   build a patch and apply it via `git apply --cached`. For untracked files,
   binary files, and unstaged deletions, stage the whole file via `git add`.

3. If no changes were applied, print `"No changes to apply"`.

4. Otherwise, print `"Applied N change(s) across M file(s)"`.

### Edge Cases

| Condition | Behavior |
|-----------|----------|
| No working-tree or index changes | Print `"No changes to stage"` and exit |
| All hunks left in their initial state then confirm | Print `"No changes to apply"` and exit |
| User cancels (q/Esc/Ctrl+C) | Print `"Cancelled"` and exit |
| Binary file in changeset | Shown as single `"(binary file)"` hunk. Toggling stages/unstages the whole file. |
| Empty untracked file | Shown as single `"(empty file)"` hunk. |
| Untracked file in subdirectory | Included via recursive status; appears under its directory group. |
| Unreadable file | Skipped with `warning: skipping unreadable file '<path>'` on stderr |
| Terminal panic during TUI | Panic hook restores terminal state before unwinding |

### Examples

```bash
# Interactive hunk selection for all changed files
$ loom add -p

# Interactive hunk selection for a specific file
$ loom add -p src/main.rs

# Interactive hunk selection using a short ID
$ loom add -p a3

# Interactive hunk selection for all (explicit)
$ loom add -p zz
```

## Target Resolution

Arguments (in both plain and `-p` modes) are resolved via `resolve_arg()`
with `accept = [File]`. See Spec 002 for the resolution algorithm.

## Prerequisites

- Must be in a git repository with a working tree (not bare)
- At least one file must have changes (staged, unstaged, or untracked)

## Design Decisions

### Bidirectional Staging Model

The TUI shows both staged and unstaged hunks together and lets the user
toggle in either direction. This was chosen over the `git add -p` model
(which only stages) because:

- Users frequently need to unstage specific hunks they previously staged
  — this eliminates the need for a separate `git reset -p`
- Showing the full picture (what is staged + what is not) makes it easier
  to reason about the final index state
- Initial selection mirrors the current index state, so confirming with no
  changes is a no-op

### Directory Grouping

Files in the left pane are grouped under directory headers. This was chosen
because repositories often have many changed files across directories, and
flat lists become hard to navigate. Directory headers also support bulk
toggle (space on a directory toggles all files within it).

### Effective Status Display

The file list shows `git status`–style two-character codes (e.g. `MM`,
`A`, `??`) that update live as the user toggles hunks. This was chosen
over simple checkmarks because:

- It mirrors what `git status` would show if the user confirmed, making the
  outcome predictable
- It distinguishes between "nothing staged" (`M`), "partially staged"
  (`MM`), and "fully staged" (`M`) at a glance

### Whole-File Operations for Special Cases

Binary files, untracked files, and file deletions cannot be partially staged
via patches. These are handled as whole-file operations (`git add` / `git
reset`) instead of hunk-level patching. They are still shown in the TUI as
single-hunk entries so the user can toggle them alongside normal hunks.
